# 1 目的と範囲

本書は DESIGN_SYSTEM.md の思想を画面遷移、画面テンプレート、利用者フローとして具体化するための公式パターン集である。  
各パターンは「エントリ」「主要画面」「遷移」「成功基準」「失敗時の回復」「レスポンシブ方針」「アクセシビリティ上の注意点」を含み、実装やコンポーネントの詳細は含まない。  
対象はオンボーディングから緊急対応、AI インタラクション、オフラインやインポート/エクスポートまで、さけLab の主要ユースケースを網羅する。

---

# 2 パターン設計の共通ルール

- **One Flow One Purpose**：各画面フローは単一の主要目的を持つ。  
- **最短経路優先**：主要タスクは最短のステップで完了できること。  
- **段階的開示**：詳細は必要に応じて段階的に提示する。  
- **回復可能性**：誤操作からの復帰手段を必ず用意する。  
- **アクセシビリティ**：キーボード操作、スクリーンリーダー、色以外の識別手段を前提とする。  
- **レスポンシブ**：モバイルファースト。緊急フローは常にモバイルでの即時操作を保証する。

# 3 アカウントと導入フロー

## 3.1 Onboarding（導入）
**エントリ**: 初回起動、アプリ紹介リンク、招待メール  
**主要画面**: ウェルカム → コア価値の短い体験（1分以内） → 最小設定（必須情報のみ） → ツアー（任意）  
**遷移**: ウェルカム → 価値体験 → 必須設定 → 完了画面（First Experience へ）  
**成功基準**: 主要機能を 60 秒以内に体験でき、初回登録率が高いこと。
  - **Success Metrics (Onboarding)**:
    - 初回主要アクション（例: ペットプロフィールの基本情報の仮登録）完了までの時間（Time to first core action）: **60秒未満**。
    - アプリ起動から会員登録完了までのオンボーディング完了率: **85%以上**。  
**失敗時回復**: 入力エラーは即時フィードバック、途中離脱時は復帰導線（メール/通知）を提供。  
**レスポンシブ**: モバイルはステップ式、デスクトップはサイドバイサイドで補助情報を表示。  
**アクセシビリティ**: 音声案内や大きなタップ領域を用意。  


## 3.2 Login（ログイン） / Signup（サインアップ）
**エントリ**: 起動、リンク、招待  
**主要画面**: メール/電話入力 → 認証（パスワード / OTP） → MFA（必要時） → 成功（Dashboard へ）  
**遷移**: Login ⇄ Forgot Password / Signup ⇄ Terms / Privacy  
**成功基準**: 認証成功で Dashboard へ遷移、失敗は明確な原因提示。  
**失敗時回復**: 認証失敗は理由と次の手順（再送、サポート）を提示。  
**レスポンシブ**: キーボード最適化、OTP 自動入力のサポート。  
**アクセシビリティ**: ラベルとエラーメッセージはスクリーンリーダーで明確に。

# 4 ダッシュボードと日常体験

## 4.1 Dashboard（ダッシュボード）
**エントリ**: ログイン後、ホームリンク、通知からの遷移  
**主要画面**: 概要サマリ（緊急アラート、今日のタスク、最近の記録） → クイックアクション（記録追加、緊急呼出）  
**遷移**: 概要 → 詳細カード → 作成フロー（CRUD）  
**成功基準**: 一目で「安心できる」主要情報が把握できること。
  - **Success Metrics (Dashboard)**:
    - ユーザーがダッシュボード上で緊急・重要アラートの存在およびステータスを認識・確認するまでの時間: **5秒未満**。
    - コア記録アクション（投薬確認・体調記録等）の完了ステップ数: **2タップ以下**。
    - ダッシュボード訪問ユーザーの継続リテンション率: 週次 **60%以上**。  
**失敗時回復**: データ取得失敗は局所的に再試行ボタンを表示。  
**レスポンシブ**: モバイルは縦スクロールで主要カードを優先表示。  
**アクセシビリティ**: 主要情報はランドマークで分離し、スクリーンリーダーで要約を提供。

## 4.2 CRUD（作成・参照・更新・削除）
**エントリ**: ダッシュボード、検索結果、詳細画面  
**主要画面**: 一覧 → 詳細 → 編集モード → 保存 / 破棄  
**遷移**: 一覧 → 新規作成（Wizard へ） / 詳細 → 編集 → 保存  
**成功基準**: 入力負担が最小で、保存後に即時反映されること。  
**失敗時回復**: 保存失敗はローカル保存と再試行、競合時は差分表示とマージ案提示。  
**レスポンシブ**: モバイルはフォームを分割して段階的に入力させる。  
**アクセシビリティ**: フォーム要素はラベルと説明を明示、必須項目は明確に。

# 5 検索・フィルタ・ナビゲーション

## 5.1 Search（検索）
**エントリ**: ヘッダー検索、ダッシュボード、一覧画面  
**主要画面**: 検索入力 → リアルタイム候補 → 結果一覧 → 結果詳細  
**遷移**: 入力 → 候補選択 → 結果 → 詳細  
**成功基準**: 目的の項目へ 3 ステップ以内で到達できること。  
**失敗時回復**: 結果がない場合は代替クエリ提案とフィルタのヒントを表示。  
**レスポンシブ**: モバイルはフルスクリーン検索、候補はタップで選択。  
**アクセシビリティ**: 検索結果は aria-live で更新を通知。

## 5.2 Filter（フィルタ）
**エントリ**: 一覧画面、検索結果  
**主要画面**: フィルタパネル（簡易 / 詳細） → 適用 → 結果更新  
**遷移**: 開閉 → 適用 / リセット  
**成功基準**: フィルタ適用で期待する結果が得られること。  
**失敗時回復**: フィルタが結果を消してしまった場合は「すべて表示」ボタンを提示。  
**レスポンシブ**: モバイルはボトムシートでフィルタを表示。  
**アクセシビリティ**: フィルタはキーボードで操作可能に。

# 6 ウィザード・設定・プロフィール

## 6.1 Wizard（ウィザード）
**エントリ**: 複雑な設定、初期セットアップ、データ移行  
**主要画面**: ステップ式画面（進捗バー） → 各ステップの検証 → 完了サマリ  
**遷移**: 次へ / 戻る / 保存して終了  
**成功基準**: ステップごとの成功率が高く、途中離脱が少ないこと。  
**失敗時回復**: 中断時は途中保存と復帰リンクを提供。  
**レスポンシブ**: モバイルは縦並びでステップを表示。  
**アクセシビリティ**: 進捗はスクリーンリーダーで説明。

## 6.2 Settings（設定）
**エントリ**: ユーザーメニュー、管理画面  
**主要画面**: アカウント、通知、プライバシー、データ管理  
**遷移**: セクション間の即時遷移、保存は局所的に行う。  
**成功基準**: 設定変更が即時反映され、元に戻せること。  
**失敗時回復**: 変更失敗は元の状態に戻し、理由を提示。  
**レスポンシブ**: セクションはアコーディオンで折りたたむ。  
**アクセシビリティ**: 設定は説明文と例を添える。

## 6.3 Profile（プロフィール）
**エントリ**: ユーザーメニュー、検索結果  
**主要画面**: プロフィール表示 → 編集モード → 保存  
**遷移**: 表示 → 編集 → 保存 / キャンセル  
**成功基準**: 情報が正確に表示され、編集が容易であること。  
**失敗時回復**: 編集失敗はローカル保存と再試行を促す。  
**レスポンシブ**: モバイルは縦積みで表示。  
**アクセシビリティ**: 画像の代替テキスト、ラベルを必須とする。

# 7 医療・記録・タイムライン

## 7.1 Medical Record（医療記録）
**エントリ**: 診療入力、病院連携、ユーザー入力  
**主要画面**: 診療サマリカード → 詳細記録 → 添付ファイル / 画像 → 更新履歴  
**遷移**: サマリ → 詳細 → 編集 → 保存 → 履歴  
**成功基準**: 出典・更新日時が明示され、誤解を生まない表示であること。  
**失敗時回復**: データ不整合は出典優先で表示し、編集履歴で差分を示す。  
**レスポンシブ**: モバイルは要約表示、詳細は展開。  
**アクセシビリティ**: 臨床用語には平易な説明を併記し、スクリーンリーダーで読み上げ可能に。

## 7.2 Timeline / History（タイムライン / 履歴）
**エントリ**: 記録追加、診療イベント、ユーザー参照  
**主要画面**: 時系列リスト → イベント詳細 → フィルタ / 検索  
**遷移**: スクロール → 詳細 → 編集 / 共有  
**成功基準**: 重要イベントが一目で分かり、振り返りが容易であること。  
**失敗時回復**: 長大な履歴はページネーションやフィルタで分割。  
**レスポンシブ**: モバイルは縦スクロール中心。  
**アクセシビリティ**: イベントはリストとして読み上げ可能に。

---

# 8 緊急・通知・オフライン

## 8.1 Emergency（緊急）
**エントリ**: 緊急ボタン、QR スキャン、外部通報リンク  
**主要画面**: 緊急カード（最短アクション） → 状況入力（最小） → 通報 / 連絡先提示 → 記録保存  
**遷移**: 緊急起動 → 主要アクション → 確認 → 完了サマリ  
**成功基準**: 主要アクションが 1〜2 タップで実行できること。緊急時の表示コピーは、感嘆符を排除した最大 60 文字以内の論理的3ステップ構造（対象＋アクション＋理由）に従うこと（文字数計算規約は `docs/foundation/BRAND_UX_AI_INTEGRATION_GUIDE.md #4.3` を参照）。
  - **Success Metrics (Emergency)**:
    - 緊急モード（QRスキャン公開画面など）における、医療書類・緊急連絡先・持病情報の初回読み込み・表示時間: **2秒未満**。
    - 緊急公開画面から緊急連絡先へのワンタップ発信、または夜間救急病院の位置（地図）表示までの操作数: **1タップ（ワンタップ）**。
    - ペット一覧カードから緊急QR公開画面を表示するまでの操作数: **2タップ以下**。  
**失敗時回復**: 通報失敗時は代替手段（SMS、QR、オフライン手順）を即提示。  
**レスポンシブ**: モバイル最優先、オフラインでも最低限の情報表示。  
**アクセシビリティ**: 大きなタップ領域、音声案内、明確なラベル。

## 8.2 Notification（通知）
**エントリ**: システムイベント、リマインダー、AI 提案  
**主要画面**: 通知一覧 → 詳細 → アクション（遷移）  
**遷移**: 通知タップ → 関連画面へ遷移  
**成功基準**: 通知が行動につながり、過剰通知を避けること。  
**失敗時回復**: 通知が誤送信された場合は取り消しと説明を提供。  
**レスポンシブ**: プラットフォーム通知とアプリ内通知の整合。  
**アクセシビリティ**: 重要通知は `aria-live="polite"` を基本とし、連続するリアルタイムデータ更新時は過剰な音声割り込みを防ぐため通知間隔制御（Debounce: 3秒）を適用する。パニック誘発を防ぐため緊急時以外の `assertive` 指定を禁止する。

## 8.3 Offline（オフライン）
**エントリ**: ネットワーク切断、低帯域  
**主要画面**: オフラインバナー → ローカル操作（記録、緊急） → 同期ステータス  
**遷移**: オフライン検知 → ローカル保存 → 同期時に解決  
**成功基準**: 重要操作がオフラインでも完了し、同期時に整合性が保たれること。  
**失敗時回復**: 同期競合は差分表示とユーザー選択で解決。  
**レスポンシブ**: オフライン UI は軽量で即時表示。  
**アクセシビリティ**: 同期状態は視覚・テキストで明示。

# 9 AI インタラクションと権限

## 9.1 AI Interaction（AI インタラクション）
**エントリ**: 提案ボタン、要約リクエスト、自動タグ付け  
**主要画面**: 提案一覧 → 詳細説明（根拠・出典） → ユーザー承認 / 編集  
**遷移**: 提案生成 → 表示 → 承認 / 編集 / 破棄  
**成功基準**: ユーザーが提案の理由を理解し、容易に編集できること。AI提案は `BRAND_UX_AI_INTEGRATION_GUIDE.md` で定める「AI出力テンプレート」に完全準拠し、ブランド感情スコア（Calm ≥ 70%, Trust ≥ 95%）を満たすこと。  
**失敗時回復**: 不適切な提案は編集・無効化・フィードバックを可能にする。  
**レスポンシブ**: 提案はコンパクトに表示し、詳細は展開。  
**アクセシビリティ**: 提案の根拠はテキストで提供し、スクリーンリーダーで読めること。

### 9.1.1 AI出力をユーザーに説明するためのテンプレート文言
AIによる提案や推測結果をユーザーに提示する際は、説明可能性（根拠・出典・信頼度）を担保するために、以下のUIテキスト表示テンプレートを使用すること。

---
#### 【AI提案表示テンプレート】
* **AI要約提案**:
  > 「[AIによる短い推奨アクション案。例: 直近の呼吸数測定値が通常より20%高いため、安静にするか夜間救急への連絡を検討してください。]」
* **提案の根拠**:
  > 本提案は、[coco]ちゃんに登録された持病（[心臓疾患]）および直近3回（[18:00, 20:00, 22:00]）の呼吸数上昇データに基づいています。
* **参照した出典/ガイドライン**:
  > - [さけLab獣医療ガイドライン v1.4](https://example.com/guideline-1.4)
  > - [日本小動物獣医師会 飼育の手引き p.45](https://example.com/handbook-45)
  > *(詳細は `docs/foundation/DESIGN_GOVERNANCE.md #19.2` 参照)*
* **確信度 / 信頼度**:
  > 確信度: **92%** (高)
* **免責事項**:
  > ※AI提案は獣医師による直接診断の代替ではありません。緊急時は直ちに専門医へご相談ください。
---

**運用参照**: DESIGN_SYSTEM.md の AI 評価メトリクスおよび DESIGN_GOVERNANCE.md の AI 監査運用指針に従い、提案のメタデータ保存と品質モニタリングを行うこと。

## 9.2 Permission（権限）
**エントリ**: 機能初回利用、外部連携、データアクセス  
**主要画面**: 権限リクエストダイアログ → 許可 / 拒否 → 設定画面で管理可能  
**遷移**: リクエスト → ユーザー選択 → 設定反映  
**成功基準**: 権限の目的と影響が明確に説明され、後で変更可能であること。  
**失敗時回復**: 拒否時は代替フローを提示し、必要時は再リクエストを案内。  
**レスポンシブ**: 権限ダイアログはモバイルでフルスクリーンに。  
**アクセシビリティ**: 権限説明は簡潔で読み上げ可能に。

# 10 システム状態と補助フロー

## 10.1 Error（エラー）
**エントリ**: 操作失敗、通信エラー、検証エラー  
**主要画面**: 局所エラーメッセージ → 詳細ヘルプ / 再試行 / サポート連絡  
**遷移**: エラー発生 → ユーザーに原因と次のアクションを提示  
**成功基準**: ユーザーが次に何をすべきか明確に理解できること。  
**失敗時回復**: 自動回復手段（再試行、ロールバック）を優先。  
**アクセシビリティ**: エラーは aria-live で通知し、スクリーンリーダーで読み上げる。

## 10.2 Empty（空状態）
**エントリ**: データ未作成、検索結果なし  
**主要画面**: 説明 + 推奨アクション（CTA） + サンプルデータ  
**遷移**: CTA → 作成フロー / チュートリアル  
**成功基準**: ユーザーが次に何をすべきか理解し、行動に移せること。  
**アクセシビリティ**: 空状態の説明は簡潔で読み上げ可能に。

## 10.3 Loading（読み込み）
**エントリ**: データ取得、同期、処理中  
**主要画面**: スケルトン / プログレスインジケータ + 状態説明  
**遷移**: 読み込み完了 → コンテンツ表示 / エラー表示  
**成功基準**: ユーザーが待ち時間を理解でき、操作不能状態が明確であること。  
**アクセシビリティ**: 読み込み状態は aria-busy 等で示す。

## 10.4 Success（成功）
**エントリ**: 保存完了、送信完了、同期完了  
**主要画面**: 成功メッセージ（短く） + 次の推奨アクション  
**遷移**: 成功 → 関連画面へ遷移または完了サマリ表示  
**成功基準**: ユーザーが操作の完了を確信できること。  
**アクセシビリティ**: 成功は aria-live で通知。

# 11 データ入出力・共有

## 11.1 QR（QR）
**エントリ**: 緊急タグ、共有リンク、オフライン参照  
**主要画面**: QR 表示 → スキャン先の最小情報 → 追加操作（連絡、保存）  
**遷移**: 表示 → スキャン → 詳細 / アクション  
**成功基準**: スキャンで必要な情報へ即時到達できること。  
**失敗時回復**: スキャン不可時は手動 URL と連絡先を提示。  
**アクセシビリティ**: QR の近くに代替テキストと手順を表示。

## 11.2 Sharing / Export / Import（共有・エクスポート・インポート）
**エントリ**: 記録共有、データ移行、バックアップ  
**主要画面**: 共有ダイアログ → 権限設定 → 送信 / ダウンロード / インポート手順  
**遷移**: 選択 → 権限設定 → 実行 → 完了サマリ  
**成功基準**: データの安全性と可搬性が確保され、ユーザーが操作を理解できること。  
**失敗時回復**: 失敗時は原因と再試行手順、サポート導線を提示。  
**アクセシビリティ**: 共有設定は読み上げ可能で、選択肢は明確に。

# 12 アクセシビリティ専用フローと回復

## 12.1 Accessibility Flow（アクセシビリティフロー）
**エントリ**: ユーザー設定、初回起動、ヘルプ  
**主要画面**: アクセシビリティ設定（フォント拡大、色反転、モーション抑制） → プレビュー → 適用  
**遷移**: 設定変更 → 即時プレビュー → 保存  
**成功基準**: 設定変更が即時に反映され、全画面で一貫すること。  
**失敗時回復**: 設定が適用されない場合はフォールバックオプションを提示。  
**アクセシビリティ**: 設定自体がアクセシブルであること。

## 12.2 Recovery（回復）
**エントリ**: 操作ミス、同期競合、データ破損  
**主要画面**: 復元ダイアログ → 選択肢（元に戻す / 上書き / マージ） → 実行  
**遷移**: 問題検知 → 回復オプション提示 → 実行 → 完了サマリ  
**成功基準**: ユーザーが安全に元の状態へ戻れること。  
**失敗時回復**: 自動バックアップとサポート連絡を提供。  
**アクセシビリティ**: 回復手順は段階的に説明し、スクリーンリーダーで追えること。

# 13 付録：パターン適用チェックリスト（短縮版）

- 主要目的は一つか。  
- 主要アクションは 3 ステップ以内か。  
- エラー時の回復手段はあるか。  
- オフライン時の最低限の機能は保証されているか。  
- AI 提案は説明可能か（要約・根拠・出典・確信度・免責・更新日時の提示）。
- 緊急時コピーは60文字以内で、感嘆符がなく、論理的3ステップ構造を満たしているか（`BRAND_UX_AI_INTEGRATION_GUIDE.md` 参照）。
- 各画面やフローのブランド感情スコア（Calm/Warmth/Trust）は目標値以上か。  
- アクセシビリティ要件（キーボード、スクリーンリーダー、色以外の識別）は満たされているか。  
- レスポンシブで主要操作が維持されるか。

# 14 将来モダリティ評価テンプレート（追加）

## 14.1 目的
音声、AR、ウェアラブル、環境センサー等の新しいモダリティを採用する際に、さけLab の体験目標（Calm / Trust / Actionability / Memory）を満たすかを一貫して評価するためのテンプレート。

## 14.2 モダリティ評価チェックリスト（採用前必須）
各項目を「合格 / 要改善 / 不合格」で評価し、合格率が 80% 以上で採用を検討する。

1. **体験目標適合**  
   - Calm: 新モダリティは利用者を落ち着かせるか。  
   - Trust: 出典・説明・透明性を担保できるか。  
   - Actionability: 最短で行動に移せるか（主要アクションが 2 ステップ以内）。  
   - Memory: 記録として残すべき情報を確実に保存できるか。

2. **情報優先度の再定義**  
   - 視覚中心の優先順位をモダリティに合わせて再定義したか。  
   - 重要情報が欠落しないか。

3. **失敗時の代替経路**  
   - モダリティが利用できない場合のフォールバック（例: 音声不可 → テキスト）を用意しているか。

4. **プライバシーとセキュリティ影響**  
   - センサーや音声データの取り扱いはプライバシー基準を満たすか。保存・転送のリスク評価を実施したか。

5. **アクセシビリティ影響**  
   - 新モダリティは既存のアクセシビリティ要件を損なわないか。支援技術との互換性を確認したか。

6. **運用コストと保守性**  
   - 長期運用に耐えうるか（保守コスト、データ保持、法令対応）。

7. **ユーザーテスト計画**  
   - 実ユーザー（支援技術利用者含む）による検証計画があるか。

## 14.3 評価結果の扱い
- **合格**（80% 以上）: パイロット導入へ進める。  
- **要改善**（50–79%）: 改善計画を作成し、再評価。  
- **不合格**（50% 未満）: 採用見送り。理由を ADR に記録。
