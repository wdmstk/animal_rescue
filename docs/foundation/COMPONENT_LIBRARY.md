# 1. 概要

本書は **COMPONENT_LIBRARY.md Version 1.0（Official）** であり、DESIGN_SYSTEM.md v2.1 の思想と責務に従って、各コンポーネントの**詳細仕様（思想的仕様）**を定義する。  
ここで定義するのは実装コードではなく、**目的・利用シーン・Do / Don't・状態一覧・アクセシビリティ要件・例外ケース・エラー時の挙動・レスポンシブ方針・アンチパターン**である。実装固有の API、トークン値、CSS、コード例は含まない。

## 1.x 色適用マトリクス（Color Application Matrix）
コンポーネントごとの推奨トークンと、通常・高コントラスト時の最低コントラスト比を以下に定義する。詳細なトークン定義は `docs/foundation/DESIGN_TOKENS.md #18.x` を参照。

| コンポーネント | 適用目的 | 推奨カラー・背景トークン | 最低コントラスト比（通常 / 高コントラスト） | WCAGレベル |
| :--- | :--- | :--- | :--- | :--- |
| **Header** | ヘッダー文字 / ヘッダー背景 | `color.text.primary` / `color.surface.header` | 4.5:1 / 7.0:1 | WCAG AA / AAA |
| **Card** | カード内テキスト / カード背景 | `color.text.primary` / `color.surface.card` | 4.5:1 / 7.0:1 | WCAG AA / AAA |
| **Alert/Banner** | 警告テキスト / 警告背景 | `color.semantic.error` / `color.surface.error` | 4.5:1 / 7.0:1 | WCAG AA / AAA |
| **Primary Action** | アクションテキスト / ボタン背景 | `color.neutral.0` / `color.action.primary` | 4.5:1 / 7.0:1 | WCAG AA / AAA |
| **Secondary Action** | アクションテキスト / ボタン背景 | `color.action.secondary` / `color.background.surface` | 4.5:1 / 7.0:1 | WCAG AA / AAA |
| **Disabled Element** | 無効化テキスト / 無効背景 | `color.text.disabled` / `color.background.disabled` | 3.0:1 / 4.5:1 | WCAG AA |

# 2. 使用上の共通ルール

- **単一責務**: 各コンポーネントは一つの明確な目的を持つ。複雑な振る舞いは複数コンポーネントの組合せで実現する。  
- **アクセシビリティ優先**: すべてのコンポーネントはアクセシビリティ要件を満たすことを前提とする。色だけで意味を伝えない。  
- **エラー回復設計**: ユーザーが誤操作しても復帰できることを前提に設計する。  
- **情報優先**: 視覚的強調は情報の優先度を反映するために使う。装飾目的の強調は避ける。  
- **レスポンシブ**: モバイルファーストを前提に、各コンポーネントは小画面での可用性を保証する。  
- **Do / Don't** は必ず守る。違反はブランド信頼を損なう。
- **ブランドトーン準拠**: すべてのコンポーネントに配置される文言、ラベル、およびボタン文言は、ブランド人格（誠実・温かい・シンプル）に完全準拠すること。エラーメッセージやラベル記述において「禁止語（最強、絶対等）」が含まれていないかをチェックする仕組みを設ける。

## 2.x オフライン同期の競合解決パターン
オフライン時にローカル保存されたデータと、サーバーに保存されたデータの同期時に競合が発生した場合は、以下のコンポーネント競合解決パターンを必ず適用する。

1. **ローカルデータの扱いと保存方針**:
   - ネットワーク接続がない場合、入力されたデータはデバイスのローカルストレージ（IndexedDB等）に即時保存され、画面上に「オフライン保存完了（未同期）」を示すインジケータ（`Feedback Components` またはトースト）を提示する。
2. **差分表示（Difference Visualization）**:
   - オンライン復帰後、サーバー側の更新日時（`updatedAt`）とローカル側の更新日時を比較し、競合が検出された場合は「競合解決ダイアログ（`Overlay Components`）」を起動する。
   - 競合箇所（例: ペットの体重、投薬履歴など）について、「サーバー側のデータ（最新）」と「ローカルのデータ（未同期）」の値を並列または上下に差分比較として明示する。
3. **マージ案提示（Merge Recommendation）**:
   - ユーザーに対して「ローカルの内容で上書きする」「サーバーの内容を維持する」「両方をマージして個別に選択して保存する」の選択肢（CTAボタン）を提示する。
   - 医療や緊急情報など、命に関わるデータの場合は、古いデータを自動で上書き消去せず、上書きされるデータを「履歴ログ」として `AuditLog` に強制退避させ、復元可能な設計とする。

## 2.y 各コンポーネントにおけるアクセシビリティ受け入れ基準
本ライブラリの各コンポーネントは、以下の受け入れ基準をすべて満たすことを必須とする。

1. **キーボード操作（Keyboard Navigation）**:
   - `Tab` キーでフォーカスが論理的な順序で移動し、`Focus Ring`（`outline: 2px solid color.action.focus` 等）が明確に視認できること。
   - インタラクティブ要素（ボタン、リンク、タブ）は `Enter` または `Space` キーで活性化・実行できること。
   - ダイアログやドロップダウンは `Esc` キーで閉じ、フォーカスが呼び出し元に戻ること。
2. **aria属性（ARIA Attributes）**:
   - 動的コンポーネント（モーダル、トグル等）は、現在の状態を `aria-expanded` や `aria-hidden` で適切にスクリーンリーダーへ通知すること。
   - 入力エラーがある場合は、`aria-invalid="true"` を設定し、`aria-describedby` でエラー内容を示すこと。
   - 主要なナビゲーションやエリアには `role` や `aria-label` を指定すること。
3. **コントラスト基準（Contrast Ratio）**:
   - 通常テキスト・アイコンと背景のコントラスト比は最低 **4.5:1**（WCAG AA）を満たすこと。
   - 高コントラストテーマ適用時は最低 **7.0:1**（WCAG AAA）を満たすこと。
4. **ユーザーテスト要件（User Testing Requirements）**:
   - 主要な支援技術（スクリーンリーダー: NVDA / VoiceOver）を使用した音声読み上げテストを実施し、情報漏れや誤読がないこと。
   - キーボードのみでの操作テストを開発サイクル内およびCI環境で行い、マウス/タッチ操作なしで全機能が完了できること。

**例外**: 実装固有のトークンは DESIGN_TOKENS.md に委譲。  
**エラー時**: 該当なし。  
**レスポンシブ**: 該当なし。  
**アンチパターン**: トークン値をここに埋め込むこと。

# 4. Navigation Components

## 4.1 Navigation（グローバルナビゲーション）
**目的**: 利用者の現在地を示し、主要な領域へ移動させる。  
**利用シーン**: アプリ全体の主要領域切替（ホーム、記録、緊急、設定等）。  
**Do**:
- 現在地を明確に示す。
- 主要アクションを優先的に配置する（緊急ボタン等）。
- モバイルでは親指で届く位置に配置する。  
**Don't**:
- 多数の項目を詰め込む。
- 重要なアクションを隠す。  
**状態一覧**: default, active, hover, focus, disabled, collapsed（モバイル）  
**アクセシビリティ**:
- 現在地は aria-current 等で明示。
- キーボードで移動可能、順序は論理的に。  
**例外ケース**:
- ゲストユーザー向けに項目を限定する場合は、主要価値を損なわないこと。  
**エラー時の挙動**:
- ナビ遷移失敗時はトーストで原因と再試行を提示。  
**レスポンシブ**:
- デスクトップ: 横並びまたはサイドバー。
- モバイル: ボトムナビまたはハンバーガーメニュー。  
**アンチパターン**:
- ナビに広告やプロモーションを混在させる。

## 4.2 Header（ヘッダー）
**目的**: 画面の目的を示し、緊急アクションやコンテキスト操作を提供する。  
**利用シーン**: 各画面の最上部。  
**Do**:
- 画面タイトルを明確に表示する。
- 必要な補助アクション（検索、戻る）を配置する。  
**Don't**:
- 情報を詰め込みすぎない。  
**状態一覧**: default, compact, sticky, hidden (immersive)  
**アクセシビリティ**:
- 見出しは h1/h2 等の適切なランドマークで提供。  
**例外**:
- 緊急フローではヘッダーを簡素化し、主要アクションを優先。  
**エラー時**:
- ヘッダー内の操作失敗は局所的にエラーメッセージを表示。  
**レスポンシブ**:
- モバイルではコンパクト化、主要アクションは優先表示。  
**アンチパターン**:
- ヘッダーに長い説明文を入れる。

## 4.3 Breadcrumb（パンくず）
**目的**: 階層構造の把握と戻る操作を支援する。  
**利用シーン**: 多層構造の画面（管理画面、詳細階層）。  
**Do**:
- 現在位置と戻る経路を明示する。  
**Don't**:
- 単純なフローで使用しない（混乱を招く）。  
**状態一覧**: default, collapsed (長いパス)  
**アクセシビリティ**:
- リスト構造で提供し、スクリーンリーダーで読み上げ可能に。  
**例外**:
- 緊急フローでは省略することが多い。  
**エラー時**: 該当なし。  
**レスポンシブ**:
- 長いパスは省略表示し、展開可能に。  
**アンチパターン**:
- パスを単なる装飾にする。

# 5. Input Components

## 5.1 Text Input（テキスト入力）
**目的**: 自由形式の短文入力を受け取る。  
**利用シーン**: 名前、メモ、検索クエリ等。  
**Do**:
- ラベルを常に表示する（プレースホルダは補助）。  
- 入力補助（フォーマット、自動補完）を提供する。  
**Don't**:
- ラベルを省略してプレースホルダだけにしない。  
**状態一覧**: default, focus, filled, error, disabled, readonly, loading  
**アクセシビリティ**:
- label と input を紐づける（for / aria-labelledby）。  
- エラーは aria-invalid と aria-describedby で説明。  
**例外ケース**:
- 緊急時の最短入力はラベルを簡素化しても良いが、後で補完を促す。  
**エラー時の挙動**:
- 具体的なエラーメッセージと修正方法を提示。  
**レスポンシブ**:
- モバイルではタップ領域を広く、キーボード最適化を行う。  
**アンチパターン**:
- エラーを赤い枠だけで示し、原因を示さない。

## 5.2 Textarea（複数行入力）
**目的**: 長文や詳細な記録を入力する。  
**利用シーン**: 日記、症状の詳細、メモ。  
**Do**:
- 文字数の目安や保存方法を示す。  
**Don't**:
- 自動的に長文を切り捨てない。  
**状態一覧**: default, focus, error, disabled, readonly, autosave  
**アクセシビリティ**:
- ラベルと説明を明示。スクリーンリーダーで行数を伝える。  
**例外**:
- 緊急時は簡易入力フォームを優先し、詳細は後で追加可能にする。  
**エラー時**:
- 保存失敗時はローカル保存と再試行を促す。  
**レスポンシブ**:
- モバイルでは折りたたみ可能、拡張可能。  
**アンチパターン**:
- 自動保存の失敗を黙って無視する。

## 5.3 Date Picker / Calendar（日付選択）
**目的**: 日付や期間を正確に選択する。  
**利用シーン**: 受診日、投薬開始日、予防接種日。  
**Do**:
- 日付フォーマットはロケールに従う。  
- 重要な日付は確認ダイアログを出す（誤入力防止）。  
**Don't**:
- 日付を自由入力だけにしない（フォーマット混乱を招く）。  
**状態一覧**: default, open, selected, disabled, error, range-select  
**アクセシビリティ**:
- カレンダーはキーボードで操作可能に。現在日・選択日を明示。  
**例外**:
- 緊急時は「今日」ボタンや「今すぐ記録」ボタンを優先。  
**エラー時**:
- 無効な日付選択は理由と修正方法を提示。  
**レスポンシブ**:
- モバイルはシンプルな日付ピッカー、デスクトップはカレンダー表示。  
**アンチパターン**:
- 日付選択を複雑な階層で隠す。

# 6. Selection Components

## 6.1 Checkbox（チェックボックス）
**目的**: 複数選択肢の選択を可能にする。  
**利用シーン**: 同意、複数タグ選択、オプション選択。  
**Do**:
- ラベルは明確に。グループラベルを提供する。  
**Don't**:
- 色だけで選択状態を示さない。  
**状態一覧**: unchecked, checked, indeterminate, disabled, focus  
**アクセシビリティ**:
- role と aria-checked を適切に設定。グループは fieldset/legend。  
**例外**:
- 緊急時の同意は最小限の情報で済ませ、後で詳細を提示。  
**エラー時**:
- 必須チェックが未選択の場合は具体的な指示を表示。  
**レスポンシブ**:
- モバイルではタップ領域を広く。  
**アンチパターン**:
- チェックボックスをトグルの代わりに使う。

## 6.2 Radio（ラジオボタン）
**目的**: 排他的な選択肢を提供する。  
**利用シーン**: 性別、単一選択の設定。  
**Do**:
- 選択肢は相互に排他的であることを明確に。  
**Don't**:
- 多すぎる選択肢を一列に並べない。  
**状態一覧**: default, selected, disabled, focus  
**アクセシビリティ**:
- fieldset/legend を使用し、ラベルを明確に。  
**例外**:
- 緊急時はデフォルトを安全側に設定することを検討。  
**エラー時**:
- 必須選択がない場合は指示を表示。  
**レスポンシブ**:
- 縦並びに切替可能。  
**アンチパターン**:
- ラジオを多段階選択に使う。

## 6.3 Switch（スイッチ）
**目的**: 二値のオン/オフを示す。  
**利用シーン**: 通知のオン/オフ、共有設定。  
**Do**:
- ラベルで意味を明確にする（状態を示すラベル）。  
**Don't**:
- スイッチを即時実行の危険な操作に使わない（確認が必要な操作）。  
**状態一覧**: on, off, disabled, loading, focus  
**アクセシビリティ**:
- role="switch" と aria-checked を使用。  
**例外**:
- 重要な切替は確認ダイアログを検討。  
**エラー時**:
- 切替失敗時は元に戻し、理由を表示。  
**レスポンシブ**:
- モバイルでのタップ領域を確保。  
**アンチパターン**:
- スイッチで複雑な設定を直接切替する。

# 7. Feedback Components

## 7.1 Toast / Snackbar（トースト / スナックバー）
**目的**: 一時的な状態通知を表示する（成功、情報、警告）。  
**利用シーン**: 保存完了、同期完了、軽微なエラー。  
**Do**:
- 短く明確なメッセージ。重要度に応じてアイコンを併用。  
- 重要な通知は自動消去しない（ユーザーが確認できるように）。  
**Don't**:
- 重要な決定をトーストだけで伝えない。  
**状態一覧**: info, success, warning, error, persistent  
**アクセシビリティ**:
- aria-live を適切に設定（重要度に応じて polite / assertive）。  
**例外**:
- 緊急時の重要通知はモーダルやバナーで代替。  
**エラー時**:
- 再試行アクションを提供する。  
**レスポンシブ**:
- モバイルでは画面下部に表示、主要操作を邪魔しない位置。  
**アンチパターン**:
- 連続して大量のトーストを表示する。

## 7.2 Banner（バナー）
**目的**: ページレベルの重要なお知らせや状態を表示する。  
**利用シーン**: メンテナンス通知、重要な同期エラー。  
**Do**:
- 重要度を明確にし、必要なアクションを提示する。  
**Don't**:
- バナーを広告に使わない。  
**状態一覧**: info, warning, critical, success  
**アクセシビリティ**:
- role="region" と aria-label で説明。  
**例外**:
- 緊急時はバナーを最上部に固定する。  
**エラー時**:
- 詳細リンクとサポート導線を提供。  
**レスポンシブ**:
- 折りたたみ可能にして画面を圧迫しない。  
**アンチパターン**:
- バナーで複数の異なる重要情報を混在させる。

# 8. Overlay Components

## 8.1 Modal / Dialog（モーダル / ダイアログ）
**目的**: 重要な決定や確認を行うためにフォーカスを集中させる。  
**利用シーン**: 削除確認、重要な設定変更、緊急確認。  
**Do**:
- 主要アクションとキャンセルを明確に配置する。  
- モーダルは最小限の情報で決断を促す。  
**Don't**:
- モーダル内に過度な情報や長文を入れない。  
**状態一覧**: open, closed, loading, error, success  
**アクセシビリティ**:
- フォーカストラップを実装し、閉じる方法を明示（Esc 等）。  
- role="dialog" と aria-modal を設定。  
**例外**:
- 緊急時はモーダルを使わず、専用の緊急 UI を優先する。  
**エラー時**:
- 操作失敗時はモーダル内で原因と次のアクションを示す。  
**レスポンシブ**:
- モバイルではフルスクリーンまたはボトムシートに切替。  
**アンチパターン**:
- モーダルの中にさらにモーダルを重ねる（多重モーダル）。

## 8.2 Drawer / Bottom Sheet（ドロワー / ボトムシート）
**目的**: 補助操作や詳細を表示し、コンテキストを保持する。  
**利用シーン**: フィルタ、詳細オプション、補助メニュー。  
**Do**:
- 閉じる操作を明確にする。  
**Don't**:
- 重要な決定をドロワーだけで行わせない。  
**状態一覧**: closed, open, partial, full, dragging  
**アクセシビリティ**:
- フォーカス管理とスクリーンリーダーでの説明を提供。  
**例外**:
- 緊急時はドロワーを使わず、主要アクションを直接提示。  
**エラー時**:
- 保存失敗時は再試行とロールバックを提示。  
**レスポンシブ**:
- モバイルでの主要パターン。デスクトップではサイドパネルに。  
**アンチパターン**:
- ドロワーに大量の設定を詰め込む。

## 8.3 Tooltip / Popover（ツールチップ / ポップオーバー）
**目的**: 補助的な説明や短いヒントを提供する。  
**利用シーン**: アイコンの説明、補助情報。  
**Do**:
- 短く明確に。表示はユーザー操作に依存する。  
**Don't**:
- 重要情報をツールチップだけで伝えない。  
**状態一覧**: hidden, visible, focus, hover  
**アクセシビリティ**:
- キーボードで表示可能にし、aria-describedby を使用。  
**例外**:
- モバイルではツールチップを代替 UI に置換する。  
**エラー時**:
- 該当なし。  
**レスポンシブ**:
- モバイルでは長押しやヘルプリンクに置換。  
**アンチパターン**:
- ツールチップに長文を入れる。

# 9. Data Display Components

## 9.1 Card（カード）
**目的**: 情報のまとまりを視覚的に提示する。  
**利用シーン**: 記録のサマリ、ペットプロフィール、医療サマリ。  
**Do**:
- 重要情報をカード上部に配置する。出典・更新日時を明示する。  
**Don't**:
- カードに過度な操作を詰め込まない。  
**状態一覧**: default, selected, loading, empty, disabled, error  
**アクセシビリティ**:
- カードはランドマークや見出しで識別可能に。  
**例外**:
- 緊急カードは視覚的に強調し、主要アクションを大きくする。  
**エラー時**:
- データ取得失敗時は再読み込みアクションを提示。  
**レスポンシブ**:
- モバイルでは縦積み、デスクトップではグリッド。  
**アンチパターン**:
- カードに複数の独立した目的を混在させる。

## 9.2 Table（表）
**目的**: 構造化された大量データを提示する。  
**利用シーン**: 診療履歴一覧、予約一覧。  
**Do**:
- 列の優先度を明確にし、重要列を左に配置する。  
- モバイルではカード表示に切替。  
**Don't**:
- すべての列を同等に扱わない。  
**状態一覧**: default, sorted, filtered, loading, empty, error  
**アクセシビリティ**:
- table semantics を正しく使用し、ヘッダとセルを紐づける。  
**例外**:
- 緊急時は表を簡素化して主要情報のみ表示。  
**エラー時**:
- データ取得失敗時は説明と再試行を提示。  
**レスポンシブ**:
- 列を折りたたむ、またはカード化。  
**アンチパターン**:
- 表に大量の操作ボタンを詰め込む。

## 9.3 Timeline（タイムライン）
**目的**: 時系列イベントを視覚化する。  
**利用シーン**: 健康記録、治療履歴。  
**Do**:
- 重要イベントを強調し、出典と日時を明示する。  
**Don't**:
- 時系列を逆に表示して混乱させない（文脈により例外あり）。  
**状態一覧**: default, highlighted, collapsed, expanded, loading, empty  
**アクセシビリティ**:
- イベントはリストとして読み上げ可能に。  
**例外**:
- 非常に長い履歴はページネーションやフィルタを提供。  
**エラー時**:
- 一部イベントの読み込み失敗は局所的に示す。  
**レスポンシブ**:
- モバイルは縦スクロール中心。  
**アンチパターン**:
- タイムラインに過度な装飾を入れる。

# 10. Layout Components

## 10.1 Grid / Container（グリッド / コンテナ）
**目的**: 画面の整列と余白を管理する。  
**利用シーン**: すべての画面レイアウト。  
**Do**:
- 一貫した余白スケールを使用する。  
**Don't**:
- 画面ごとに独自のグリッドを乱用しない。  
**状態一覧**: responsive breakpoints（概念）  
**アクセシビリティ**:
- レイアウトは読み上げ順を乱さない。  
**例外**:
- 緊急フローは単純なレイアウトを優先。  
**エラー時**:
- レイアウト崩れはフォールバックレイアウトを用意。  
**レスポンシブ**:
- モバイルファーストで設計。  
**アンチパターン**:
- グリッドを視覚的装飾のために乱用する。

## 10.2 Stack / Spacer（スタック / スペーサー）
**目的**: コンポーネント間の一貫した間隔を提供する。  
**利用シーン**: コンポーネントの縦横配置。  
**Do**:
- スペーサーで意味のある余白を作る。  
**Don't**:
- マージンを個別に乱用しない。  
**状態一覧**: small, medium, large（概念）  
**アクセシビリティ**:
- 余白は読み上げに影響しない。  
**例外**:
- 緊急時は余白を縮めて情報を優先する。  
**エラー時**: 該当なし。  
**レスポンシブ**:
- 余白スケールはブレークポイントに応じて変化。  
**アンチパターン**:
- スペーサーでレイアウトロジックを複雑化する。

# 11. Identity Components

## 11.1 Avatar（アバター）
**目的**: 人やペットの識別を視覚的に補助する。  
**利用シーン**: プロフィール、コメント、記録の所有者表示。  
**Do**:
- 名前と組み合わせて表示する。代替テキストを必ず提供。  
**Don't**:
- アバターだけで重要な識別情報を伝えない。  
**状態一覧**: default, loading, fallback, offline, verified  
**アクセシビリティ**:
- alt テキストや aria-label を提供。  
**例外**:
- 匿名データは識別情報をマスクする。  
**エラー時**:
- 画像読み込み失敗時はイニシャルやプレースホルダを表示。  
**レスポンシブ**:
- サイズをスケールして表示。  
**アンチパターン**:
- アバターに重要な操作を割り当てる（誤タップの原因）。

## 11.2 Profile Card（プロフィールカード）
**目的**: 人やペットの主要情報をまとめて提示する。  
**利用シーン**: ペット詳細、飼い主情報、担当医情報。  
**Do**:
- 主要情報（名前、年齢、注意点）を上部に配置。出典と更新日時を明示。  
**Don't**:
- 個人情報を不必要に公開しない。  
**状態一覧**: default, editable, loading, empty, error  
**アクセシビリティ**:
- 見出しとランドマークで識別可能に。  
**例外**:
- プライバシー設定により一部情報を隠す。  
**エラー時**:
- 情報取得失敗時は再読み込みとサポート導線を提示。  
**レスポンシブ**:
- モバイルは縦積みで表示。  
**アンチパターン**:
- プロフィールカードに広告や外部リンクを混在させる。

# 12. Medical Components

> 医療情報は特に慎重に扱う。誤解を招かない表現、出典の明示、更新日時の表示を必須とする。

## 12.1 Medical Record Card（医療記録カード）
**目的**: 診断、処方、検査結果などの医療情報を要約して提示する。  
**利用シーン**: 診療履歴、受診サマリ。  
**Do**:
- 出典（誰が記録したか）、更新日時、重要度を明示する。  
- 臨床的な用語は平易な説明を併記する。  
**Don't**:
- 推測を断定として表示しない。  
**状態一覧**: default, critical, reviewed, pending, archived, error  
**アクセシビリティ**:
- 構造化データとして読み上げ可能に。重要情報は aria-live で通知。  
**例外**:
- 機微な医療情報はアクセス制御を厳格にする。  
**エラー時**:
- データ不整合時は出典を優先し、編集履歴を提示。  
**レスポンシブ**:
- モバイルでは要約表示、詳細は展開。  
**アンチパターン**:
- 医療情報を色だけで示す。

## 12.2 Medication Card（投薬カード）
**目的**: 投薬名、用量、投与スケジュールを明確に提示する。  
**利用シーン**: 投薬管理、リマインダー。  
**Do**:
- 用量・頻度・開始日・終了日・注意事項を明示。  
- 服薬アラートは明確で非脅迫的に。  
**Don't**:
- 投薬情報を省略して表示しない。  
**状態一覧**: active, paused, completed, missed, error  
**アクセシビリティ**:
- 服薬手順は読み上げ可能に。  
**例外**:
- 緊急時は最小限の服薬情報を優先表示。  
**エラー時**:
- リマインダー失敗時は再通知と手動確認を促す。  
**レスポンシブ**:
- モバイルでの確認操作を簡素化。  
**アンチパターン**:
- 投薬カードに広告や非関連情報を混在させる。

## 12.3 Vaccination Card（予防接種カード）
**目的**: 接種履歴と次回予定を提示する。  
**利用シーン**: 予防接種管理、渡航時の確認。  
**Do**:
- 接種日、ワクチン名、ロット番号、接種者を明示。  
**Don't**:
- 不確かな接種情報を断定的に表示しない。  
**状態一覧**: up-to-date, due, overdue, exempt, error  
**アクセシビリティ**:
- 履歴は時系列で読み上げ可能に。  
**例外**:
- 法的要件がある場合は表示方法を調整。  
**エラー時**:
- データ不整合は出典確認を促す。  
**レスポンシブ**:
- モバイルでの確認を容易に。  
**アンチパターン**:
- 接種情報を視覚的に煽る表現。

# 13. Emergency Components

> 緊急コンポーネントは即時性と明確さを最優先とする。誤操作を防ぐための確認と、オフラインでの最低限の機能を保証する。

## 13.1 Emergency Card（緊急カード）
**目的**: 緊急時に最短で必要な情報と行動を提示する。  
**利用シーン**: 緊急通報、救急対応時の情報提示。  
**Do**:
- 主要アクション（通報、連絡先、最短手順）を一目で示す。  
- 状況把握に必要な最小限の情報を優先表示。  
**Don't**:
- 補助情報を優先して表示しない。  
**状態一覧**: idle, active, acknowledged, resolved, offline  
**感情・評価**: 緊急コンポーネントは、Calm Score（落ち着き度）を極大化（目標 90% 以上）するデザイン・文言とし、利用者の心理的パニックを回避するように構成する。
**アクセシビリティ**:
- 主要アクションはキーボードで即時操作可能に。音声案内や大きなタップ領域を考慮。  
**例外**:
- オフライン時はローカルに保存された連絡先と手順を提示。  
**エラー時**:
- 通報失敗時は代替手段（SMS、QR）を提示し、再試行を促す。  
**レスポンシブ**:
- モバイルでの即時操作を最優先。  
**アンチパターン**:
- 緊急カードに広告やプロモーションを含める。

## 13.2 QR Emergency Card（QR 緊急カード）
**目的**: スキャンで緊急情報へアクセスさせるための QR 表示。  
**利用シーン**: 首輪タグ、ステッカー、印刷物。  
**Do**:
- QR が指す情報は最小限かつ安全に設計する（公開情報と限定情報の区別）。  
- オフラインでのアクセス方法を明示する。  
**Don't**:
- QR に過剰な個人情報を埋め込まない。  
**状態一覧**: visible, hidden, expired, invalid  
**アクセシビリティ**:
- QR の近くに短い説明文と代替手段を提供。  
**例外**:
- 紙媒体では QR の有効期限を明示する。  
**エラー時**:
- スキャン失敗時は手動 URL と連絡先を提示。  
**レスポンシブ**:
- 表示サイズは印刷・画面双方で読み取りやすく。  
**アンチパターン**:
- QR を小さくして読み取り困難にする。
