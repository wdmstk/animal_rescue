# 商用化ガイド（運用・セキュリティ）

本書は `animal_rescue` の商用運用に向けた実務Runbookです。

## 1. セキュリティ基盤

### データ保護
- 通信: TLS強制
- 保存: DB/バックアップ/オブジェクトの暗号化
- 権限: 最小権限（Supabase RLS + サービスロール限定使用）
- 機微情報: ログに個人情報/秘密情報を出力しない

### 秘密情報管理
- `.env` はローカル限定
- 本番鍵はシークレットマネージャで管理
- 四半期ローテーション（Stripe secret, Supabase service role等）

### 脆弱性管理
- 依存ライブラリ監視（週次）
- 脆弱性診断（四半期）
- ペネトレーションテスト（年次）

## 2. Supabase / Stripe連携チェック

### Stripe Webhook
- 署名検証を必須化
- イベント処理の冪等化（重複イベント対策）
- 失敗時リトライと死活監視
- イベント監査ログ（event id, status, timestamp）を保存

### 決済データ責任分界
- カード情報はStripeホスト型UIで処理し、自社保持しない
- 自社保持は顧客ID/サブスクID/ステータスなど最小メタデータのみ

## 3. 監視・アラート

### 監視対象
- 可用性: API成功率、5xx率、レイテンシ
- セキュリティ: 認証失敗急増、異常アクセス
- 課金: checkout失敗率、webhook遅延/失敗
- 公開機能: 緊急公開APIのエラー率

### アラート設計
- Sev1: 全体停止/課金不能/公開機能障害（即時通知）
- Sev2: 部分障害/性能劣化（営業時間内即時）
- Sev3: 軽微異常（定時確認）

## 4. BCP / DR（災害復旧）

### 目標
- RTO: 4時間
- RPO: 1時間

### 復旧優先順位
1. 認証
2. 緊急公開API
3. ペット健康記録の閲覧
4. 更新系API
5. 管理/分析系

### 手順
- 定期バックアップ取得
- 月次リストア演習
- 障害時Runbook（判断者、連絡先、復旧コマンド）整備

## 5. インシデント対応

### 初動
1. 検知（監視/通報）
2. 影響判定（ユーザー範囲・データ範囲）
3. 封じ込め（鍵ローテーション、機能制限）
4. 一次告知

### 事後
- 根本原因分析（RCA）
- 恒久対策のIssue化
- 再発防止レビュー

## 6. サポート / SLA
- 問い合わせ窓口（メール/フォーム）
- FAQ整備（課金、解約、公開リンク、データ削除）
- 障害告知ページ運用
- SLAを提供する場合は稼働率と補償条件を規約と整合

## 7. PCI DSS実務
- Stripeホスト型決済を維持
- SAQ種別を確認し、年次チェック証跡を保管
- 決済関連アクセス権限を最小化し、監査ログを保持
