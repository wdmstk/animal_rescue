## 1. インフラ構成

### 現在のインフラ

```
[ユーザー]
   ↓ HTTPS
[Vercel]
   ├── Edge Network（CDN・静的コンテンツ）
   ├── Serverless Functions（Next.js API Routes）
   └── Build Pipeline（CI/CD）
   ↓
[Supabase]
   ├── PostgreSQL（プライマリDB）
   ├── Supabase Auth（認証・JWT）
   ├── Supabase Storage（写真・書類）
   └── Edge Functions（必要に応じて）
   ↓
[External Services]
   ├── Stripe（決済）
   └── OCR API（医療書類読取）
```

### 環境構成

| 環境 | 用途 | DB | 注意 |
|---|---|---|---|
| Local | 開発 | ローカルSupabase | .env.local |
| Test | CI・統合テスト | テスト用Supabase | 本番データなし |
| Staging（未整備） | リリース前確認 | 本番同等DB | 設計中 |
| Production | 本番 | 本番Supabase | 厳格な権限管理 |

**問題**: Staging環境が未整備
**改善案**: Vercel Preview Deployment + Supabase Branch DB（Supabase Pro以上）

## 2. デプロイ設計

### CI/CDパイプライン（GitHub Actions）

```
PR作成:
  1. Lint check（npm run lint）
  2. Unit / Integration Tests（npx vitest run）
  3. DB Integration Tests（Real Postgres）
  4. E2E Tests（Playwright）
  5. PR Pre-Merge Guard（チェックボックス確認）

mainマージ後:
  1. Vercel が自動デプロイ（Production）
  2. Post-merge CI確認
  3. 対応Issue のクローズ
  4. TASKS.md の更新
```

### デプロイ判断基準

```
マージ条件（全て必要）:
  - lint: green
  - unit/integration tests: green
  - E2E tests: green
  - セルフレビュー完了
  - 対応Issueがクローズされている
  - PR テンプレートのチェック完了
```

### ロールバック手順

```
方法1: Vercel の即時ロールバック
  - Vercel ダッシュボード → Deployments → 前のデプロイを「Promote」
  - 所要時間: 1〜2分

方法2: git revert + Push
  - `git revert <commit>` → PR → マージ
  - 所要時間: 10〜30分（レビュー含む）

DBマイグレーションのロールバック:
  - Prisma: 逆マイグレーションファイルの作成（手動）
  - Supabase PITR: 特定時点への復元
  - 重要: DBの下位互換性を維持するマイグレーション設計が必須
```

## 3. 監視・アラート設計

### 監視項目と閾値

```
インフラ監視:
  - 可用性: HTTP 200 レスポンス確認（1分ごと）
  - API p95レスポンスタイム: > 2000ms でアラート
  - 5xx エラーレート: > 0.5% でアラート
  - Supabase 接続エラー: 発生でアラート

ビジネス監視:
  - 新規登録数の急落（前日比50%以下）
  - 課金エラー率: > 5% でアラート
  - 緊急公開API エラー率: > 1% でアラート（最重要）

セキュリティ監視:
  - 認証失敗の急増: 5分で100回超でアラート
  - 同一IPからの異常アクセス: 1分で300リクエスト超
```

### ツール（推奨）

```
APM・エラー追跡:
  Sentry（推奨）
  - エラーの追跡・スタックトレース
  - パフォーマンスモニタリング
  - セッションリプレイ（UX改善）

可用性監視:
  UptimeRobot または Better Uptime（推奨）
  - エンドポイントの死活監視
  - ステータスページ

ログ集約:
  Vercel の組み込みログ（当面）
  → 規模拡大時に Datadog または Grafana Loki

ダッシュボード:
  管理画面の KPI ダッシュボード（独自実装）
  + Google Analytics（匿名化されたアクセス解析）
```

## 4. 障害対応設計

### 障害レベル定義

```
Sev1（最重要）: サービス全体停止・緊急公開API障害
  → 即時対応開始（24時間365日）
  → ユーザー通知: ステータスページ + メール
  → 初動目標: 30分以内

Sev2（重要）: 部分機能障害・課金機能停止
  → 営業時間内に即時対応（深夜は翌朝）
  → ユーザー通知: ステータスページ
  → 初動目標: 2時間以内

Sev3（軽微）: パフォーマンス劣化・軽微なバグ
  → 翌営業日に対応
  → ユーザー通知: 不要
  → 初動目標: 翌営業日
```

### 障害対応フロー

```
1. 検知（監視アラート / ユーザー報告）
   ↓
2. 影響範囲の特定
   - 何の機能が影響を受けているか
   - 何人のユーザーが影響を受けているか
   ↓
3. 初動対応
   - Sev1: 必要なら機能の一時停止・ロールバック
   - ステータスページの更新
   ↓
4. 根本原因の調査
   - ログの確認
   - Sentryのエラートレース
   ↓
5. 修正・デプロイ
   - 緊急パッチの作成・テスト・デプロイ
   ↓
6. 復旧確認
   - 全テストの実施
   - ユーザーへの復旧報告
   ↓
7. 事後分析（ポストモーテム）
   - 根本原因の特定
   - 再発防止策のIssue化
   - 対応手順書の更新
```

### 緊急公開API障害時の対応

```
本機能は救急時に利用されるため、最優先で復旧する。

対応手順:
  1. エラーを検知（監視アラート）
  2. Supabase ダッシュボードで RPC の状態確認
  3. Vercel Functions のエラーログを確認
  4. DB接続が問題の場合: Supabase サポートへ連絡
  5. コードの問題の場合: 即時ロールバック

ユーザーへの通知:
  - ステータスページで障害情報を公開
  - 「緊急情報の印刷版をご用意ください」の代替手段を案内

代替手段の案内:
  - アプリ内の「PDF出力」機能での事前印刷を推奨
  - QRコードシール（将来機能）の重要性の説明
```

### インシデントコミュニケーション

```
ステータスページ: https://status.{domain}/
  - 障害発生時: 即時更新（5分以内）
  - 復旧時: 復旧報告と影響範囲の説明
  - 保守作業: 24時間前から予定告知

ユーザーへのメール:
  - Sev1で1時間以上継続の場合
  - 個人情報漏洩の場合（必須・法的義務）
```

## 5. バックアップ・DR設計

### バックアップ計画

```
Supabase の自動バックアップ:
  - 日次フルバックアップ（有料プランで有効化）
  - PITR: 連続バックアップ（Pro プラン）
  - 保持期間: 7日間

ファイルストレージ（写真・書類）:
  - Supabase Storage は内部で S3 互換
  - 自動バックアップは Supabase に依存
  - 重要なファイルは別リージョンへの定期コピーを推奨

バックアップ確認:
  - 月次リストア演習（本番データをStaging環境に復元）
  - 確認項目: リストア所要時間・データ整合性
```

### DR（災害復旧）計画

```
目標:
  RTO: 4時間
  RPO: 1時間

シナリオと対応:
  1. Vercel 障害
     → 別のホスティング（Railway / Netlify）への一時移行
     → Vercel のSLA 99.99%のため確率は低い
  
  2. Supabase 障害
     → Supabase の Status Page を確認
     → データは別リージョンの Supabase に PITR で復元
     → 接続文字列の切替（環境変数変更）
  
  3. Stripe 障害
     → 課金機能のみ停止（他機能は継続）
     → ユーザーへの通知（「課金処理が一時停止しています」）
     → Stripe の公式ステータスを監視
  
  4. 大規模データ破損
     → PITR で損失前の時点に復元
     → 影響を受けたユーザーへ個別通知
     → 必要に応じて個人情報保護委員会への報告
```

## 6. 運用定例

### 日次

```
□ エラー監視ダッシュボードの確認
□ Sentry のエラー件数確認
□ Stripe のWebhookエラー確認
□ 問い合わせメールの確認・返答
```

### 週次

```
□ KPI レビュー（登録数・有料転換率・解約数）
□ エラーログの振り返り
□ セキュリティアラートの確認
□ 依存パッケージのアップデート確認（npm audit）
□ バックアップの確認
```

### 月次

```
□ 月次KPI分析（MRR・チャーン・NPS）
□ 解約理由の分析
□ 月次リストア演習
□ セキュリティ定期確認（RLSポリシー差分・権限）
□ コスト確認（Vercel・Supabase・Stripe手数料）
□ ロードマップの更新
```

### 四半期

```
□ セキュリティ診断（脆弱性スキャン）
□ 秘密情報のローテーション
□ 競合調査・市場分析
□ ロードマップの大幅見直し
□ ペネトレーションテスト（年次）
□ 利用規約・プライバシーポリシーの見直し
```

## 7. サポート設計

### サポート窓口

```
メール: support@{domain}
  - 受付: 24時間（返答は営業時間内）
  - 初回返答: 24時間以内
  - 解決目標: 72時間以内

FAQ（Webページ）:
  必須項目:
  - 解約方法
  - 返金について
  - データの削除方法
  - パスワードを忘れた場合
  - 家族を招待する方法
  - QRコードが表示されない場合
  - 緊急公開URLが開けない場合
```

### サポートの分類と対応

```
課金・請求（最多問い合わせ予想）:
  - 解約: Stripe Customer Portal の使い方を案内
  - 返金: 方針に基づき個別対応
  - 請求エラー: Stripe ダッシュボードで確認

技術的問題:
  - 再現手順を確認
  - Sentry でエラーログ確認
  - 必要ならデバッグ用のアカウントで確認

データ関連:
  - 開示請求: 本人確認後 → 14日以内に対応
  - 削除請求: 本人確認後 → 削除スケジュール確認
  - 訂正請求: 本人確認後 → 手動修正またはUI案内
```
