## 2. 未解決課題一覧

### UQ-01: オフライン対応の方針決定

**課題**: 通信障害時（山間部・地下・トンネル）に緊急QR画面が表示できない
**影響**: 救急現場での最悪ケースで機能しない
**選択肢**:
  A. Service Worker キャッシュ（推奨）
  B. 印刷PDF推奨（短期対応として有効）
  C. 何もしない（リスク許容）
**決定ステータス**: **決定（A+Bのハイブリッド）**
**方針**: 短期的に「印刷PDF出力機能」の導線を強化し、Phase 2にてPWA化およびService Workerによる緊急情報のローカルキャッシュ（暗号化ストレージ保管）を実装する。

---

### UQ-02: 緊急情報の構造化設計

**課題**: 現状の持病・アレルギーはフリーテキストのため、緊急時の情報提示品質が低い
**影響**: 医療従事者が優先度の高い情報を素早く見つけられない
**選択肢**:
  A. タグ入力形式（既存フリーテキストと並存）
  B. 完全なフリーテキスト廃止（過去データ変換必要）
  C. 構造化 + フリーテキストの両方を維持
**決定ステータス**: **決定（Aの並存アプローチ）**
**方針**: 既存のフリーテキスト入力を維持して入力負荷を下げつつ、DBスキーマに `PetEmergencyInfoTag` を追加。UI上でよくあるアレルギー（例: ペニシリン系、鶏肉等）をワンタップでタグとして追加し、緊急画面で赤色バッジ表示可能にする。

---

### UQ-03: 管理画面の認証設計

**課題**: 管理者ユーザーの区別・認証方法が未設計
**影響**: 管理画面の実装が始められない
**選択肢**:
  A. Supabase Auth のロール拡張（admin ロール）
  B. 別認証システム（Basic Auth + IP制限）
  C. Supabase Auth + HouseholdMember とは別のadmin テーブル
**決定ステータス**: **決定（Aのカスタムクレーム方式）**
**方針**: Supabase Authの `app_metadata` に `role: "admin"` を付与し、APIルートのミドルウェアおよびRLSポリシーで `auth.jwt() ->> 'role' = 'admin'` の検証を行う。

---

### UQ-04: プッシュ通知の実装方法

**課題**: 現在はメール通知のみ。Web Push / ネイティブアプリのどちらで対応するか
**影響**: 投薬リマインダーの実効性・ワクチン期限通知
**選択肢**:
  A. Web Push（PWA）
  B. LINE Notify / LINE Messaging API
  C. ネイティブアプリ（React Native）
  D. メールのみで継続
**決定ステータス**: **決定（AのWeb Pushアプローチ）**
**方針**: PWAをベースとした「Web Push通知」を標準採用する。Phase 1ローンチ段階はメール送信のみとし、Phase 2にてPWA化と同時にWeb Push（プッシュ通知API連携）を実装する。

---

### UQ-05: ペット死亡後のデータ管理

**課題**: ペットが亡くなった場合、飼い主がアカウントを解約するケースが多い（最大の解約トリガー）
**影響**: 解約率・LTV
**選択肢**:
  A. アーカイブモード（「思い出として残す」）
  B. 削除のみ
  C. 有料の「思い出プラン」
**決定ステータス**: **決定（Aのアーカイブ・思い出保管）**
**方針**: アカウント削除ではなく、ペットデータを読み取り専用として残す「アーカイブモード」を実装。データ量が制限された安価な「思い出保管プラン（例: 月額100円）」またはそのまま無料閲覧制限として維持することで、感情的な繋がりを保ちチャーン率を抑える。

---

### UQ-06: 多頭飼いの体重・検査値比較

**課題**: 複数ペットを飼っている場合、ペット間のデータ比較が難しい
**影響**: ユーザー体験・データ価値
**選択肢**:
  A. ペット間比較グラフ機能を追加
  B. 現状維持（ペット単位での管理）
**決定ステータス**: **決定（Phase 2にてAを追加）**
**方針**: Phase 1（初期リリース）は現状維持（ペット詳細画面での個別表示）とする。Phase 2において、世帯内の同種ペット間で体重や検査項目を並列比較できる比較分析ダッシュボード機能を追加する。

---

### UQ-07: 動物病院との連携モデル

**課題**: 将来的な動物病院連携のためのビジネスモデルと技術設計が未定
**影響**: B2B展開・スケールアップ
**必要な決定**:
  - 病院がユーザーに見せる情報の範囲
  - 病院が情報を書き込む権限の有無
  - 病院向け料金プラン
  - 技術的なAPI設計
**決定ステータス**: **Phase 2の課題**

## 4. AI活用提案（補助AI）

### 許可される AI 活用（医療判断を含まない補助機能）

```
1. OCR（実装済み・TASK-185）
   目的: 医療書類・領収書のテキスト抽出
   実装: 医療書類写真 → OCR → テキスト確認 → 記録保存
   制約: OCRの結果は必ずユーザーが確認・編集してから保存

2. 薬剤名読取補助（強化余地あり）
   目的: 薬袋・処方箋から薬剤名を自動入力
   実装: OCR + 薬剤名データベースとのマッチング
   制約: 用量・副作用の自動入力は禁止

3. ワクチン証明書読取
   目的: ワクチン証明書から接種日・種別を自動入力
   実装: OCR + 構造化データ抽出
   制約: 読取結果は必ず確認

4. 領収書・診療明細読取
   目的: 診療費の記録を自動化
   実装: OCR + 金額・日付の抽出
   制約: 医療内容の解釈は行わない

5. カルテ要約（将来・高難易度）
   目的: 長い診察記録をサマリー化
   実装: LLM による要約
   制約: 要約は必ず「ユーザーの入力した内容のまとめ」であることを明示
          「診断」「治療方針」の提案は禁止

6. 症状メモ整理（将来・高リスク）
   目的: 「元気がない・食欲不振」を構造化
   実装: 症状カテゴリへの分類
   リスク: 「この症状は○○の可能性があります」という表現を避ける
   制約: 分類は「記録の整理」のみ。診断的文言は禁止

7. 病院検索支援（将来）
   目的: 近くの動物病院を検索・表示
   実装: Google Maps API + ペット種類でのフィルター
   制約: 「おすすめ病院」「評判の良い病院」の提示は慎重に（景品表示法）
```
