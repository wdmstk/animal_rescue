## 2. UXガイドライン

### 認知負荷最小化の原則

#### 通常時のUX
- **情報の段階的開示**: 最重要情報をファーストビューに、詳細はスクロールで
- **明確なCTA**: 1画面に1つの主要アクションを原則とする
- **進捗の可視化**: オンボーディング・ロード状態は常にフィードバックする

#### 緊急時のUX（最重要）

**設計原則**:
1. **Zero Thinking**: 考えなくても操作できる
2. **Large Touch Target**: 震えた手でも操作できる（48px以上）
3. **High Contrast**: 暗い環境でも読める（コントラスト比 4.5:1以上）
4. **Fast Loading**: 3G回線でも3秒以内に表示

**緊急時の3タップルール**:
```
タップ1: アプリのアイコンをタップ → ペット一覧が開く（ログイン維持前提）
タップ2: ペットをタップ → ペット詳細が開く
タップ3: 「緊急」ボタンをタップ → QRコードが全画面表示
```

**現状の問題**:
- ログインセッションが切れている場合、タップ数が増える
- **改善案**: 生体認証による素早い再ログインを実装

#### ユーザーフローのストレスポイント

| フェーズ | ストレスポイント | 改善案 | 優先度 |
|---|---|---|---|
| 初回登録 | フォーム項目が多い | プログレッシブな入力（最小限で開始） | High |
| ペット情報入力 | 品種の選択肢が多い | テキスト入力 + オートコンプリート | Medium |
| 緊急情報入力 | フリーテキストで構造化されていない | タグ入力・チェックリスト形式 | High |
| 緊急時QR表示 | QRボタンが下にある | スティッキーボタン配置 | High |
| 緊急情報閲覧 | 情報が並列で優先度不明 | 重要度でソート・色分け | High |
| 課金導線 | トライアル終了後の突然の制限 | 30日前・7日前・3日前の通知 | Medium |
| 解約 | 解約が分かりにくい | 設定画面に明確な解約ボタン | Must（特商法） |

#### 離脱ポイントと対策

| 離脱ポイント | 推定離脱率 | 対策 |
|---|---|---|
| 登録時の入力途中 | 高（推定30%） | 自動保存・「あとで入力」の提供 |
| 緊急情報未入力のまま放置 | 高（推定50%） | オンボーディング完了バッジ・督促通知 |
| トライアル終了 | 高（推定60%） | 価値を実感させる体験の設計 |
| 操作エラー時 | 中（推定15%） | わかりやすいエラーメッセージ・リトライ |

**目的**: 既存ユーザーのログイン

**レイアウト**:
```
┌─────────────────────────────────┐
│  [ロゴ・サービス名]              │
│  「動物の救急手帳」              │
│                                  │
│  [メールアドレス入力]            │
│  [パスワード入力]                │
│                                  │

│                                  │
│  [パスワードを忘れた方]  ← 未実装│
│  ─────────────                  │
│  [新規登録へ]                   │
└─────────────────────────────────┘
```

**設計要件**:
- フォームバリデーションはリアルタイム（onChange）ではなく、blur時に実施
- エラーメッセージ: 「メールアドレスまたはパスワードが正しくありません」（詳細は開示しない）
- ログイン成功後: `/pets` へリダイレクト
- 既にログイン済みの場合: `/pets` へ自動リダイレクト

**未実装課題**:
- パスワードリセット機能（FR-01-07）
- ソーシャルログイン（Google/LINE）

## 4. 緊急時UX設計

### 3タップ到達設計

```
現在の到達パス:
  アプリ起動 → ペット一覧 → ペット詳細 → QRボタン = 3タップ ✅
  （ただし: QRボタンがページ下部にある場合はスクロールが必要）

改善後の到達パス:
  アプリ起動 → ペット一覧 → ペットの「緊急」ボタン = 2タップ ✅✅
```

### 通信障害時の対応（重要な未解決課題）

**現状**: 完全にオンライン依存。通信障害時は緊急公開画面にアクセス不可。

**問題の深刻さ**: 山間部・地下・トンネルなど通信が不安定な場所での緊急事態では機能しない。

**解決策の選択肢**:

| 方針 | 実装難易度 | UX | セキュリティ |
|---|---|---|---|
| A. Service Worker キャッシュ（緊急情報） | Medium | 良好 | 要注意（端末紛失時） |
| B. 印刷済みQR + ステッカー | Low | 制限あり | 低リスク |
| C. オフライン対応PWA | High | 最良 | 要設計 |
| D. ロック画面ウィジェット（iOS/Android） | Very High | 最良 | 端末認証で保護 |

**推奨**: まず B（印刷機能は実装済み）を推進しつつ、A を Phase 2 で実装

### QRコードの品質要件

```
QRコード仕様:
  バージョン: 自動（データ量に応じて）
  エラー訂正レベル: M（30%欠損まで復元可）
  サイズ: 最小250px × 250px
  カラー: 黒（#000000）on 白（#FFFFFF）
  マージン: 4モジュール以上

表示環境:
  モバイル画面: 全画面表示
  印刷: A4紙で4cm × 4cm以上
  首輪タグ: 別途QRコードシール対応（将来機能）
```

## 5. アクセシビリティ設計

### WCAG 2.1 AA準拠要件

| 基準 | 要件 | 現状 | 対応状況 |
|---|---|---|---|
| 1.1.1 | 非テキストコンテンツに代替テキスト | 写真にalt属性 | TASK-188で対応 |
| 1.3.1 | 情報と関係性の構造化 | セマンティックHTML | 要確認 |
| 1.4.1 | 色だけに依存しない情報伝達 | アイコン+色の併用 | 要確認 |
| 1.4.3 | コントラスト比 4.5:1以上 | カラーパレット設計 | TASK-188で対応 |
| 2.1.1 | キーボード操作 | フォーカス管理 | 部分対応 |
| 2.4.3 | フォーカス順序 | タブ順序の最適化 | 要確認 |
| 3.3.1 | エラー識別 | エラーメッセージの明確化 | TASK-187で対応 |

### 高齢者・緊急時の特別配慮

```
文字サイズ:
  通常画面: 16px以上
  緊急画面: 20px以上
  見出し: 24px以上

タッチターゲット:
  通常ボタン: 44px以上（TASK-189で対応）
  緊急ボタン: 56px以上
  発信ボタン: 64px以上（全幅）

コントラスト:
  通常テキスト: 4.5:1以上
  大きいテキスト: 3:1以上
  緊急情報: 7:1以上（AAA）

動作:
  アニメーション: prefers-reduced-motion を尊重
  フラッシュ: なし
  自動音声: なし
```

## 6. レスポンシブ設計

### ブレークポイント

```
Mobile:  375px - 767px  （メイン対象）
Tablet:  768px - 1023px （iPad等）
Desktop: 1024px以上     （副次的）
```

### モバイルファーストの原則

全コンポーネントはモバイルサイズで設計し、デスクトップで拡張する。

```
モバイル（375px）:
  - 1カラムレイアウト
  - ナビ: ボトムナビゲーション
  - ペットカード: フルワイド

タブレット（768px）:
  - 2カラムレイアウト（ペットリスト）
  - ナビ: サイドバー（任意）

デスクトップ（1024px+）:
  - 最大幅: 1200px
  - コンテンツ中央寄せ
  - ナビ: サイドバー固定
```

### スワイプジェスチャー

```
ペット詳細画面:
  ← スワイプ: 前のペット
  → スワイプ: 次のペット

QR画面:
  下スワイプ: 閉じる（フルスクリーンから戻る）

緊急公開画面:
  スワイプなし（意図しない操作を防ぐ）
```

### 却下した設計案

**案A: ネイティブアプリ開発**
- 採用理由: コスト削減のためPWAで対応
- 却下理由: ネイティブアプリの開発・維持コストが大きい
- トレードオフ: ロック画面ウィジェット等の深いOS統合は不可

**案B: チャット形式の情報入力**
- 採用理由: LINEユーザーに親しみやすい
- 却下理由: 緊急情報の構造化が難しい
- トレードオフ: UXは良いが、データの正確性に課題
