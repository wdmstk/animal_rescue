## 1. パフォーマンス要件

### レスポンスタイム目標

| エンドポイント / 操作 | 目標 (p50) | 目標 (p95) | 目標 (p99) | 優先度 |
|---|---|---|---|---|
| 公開緊急API `GET /api/public/emergency/:token` | < 200ms | < 500ms | < 1000ms | Critical |
| ペット一覧 `GET /api/pets` | < 300ms | < 800ms | < 2000ms | High |
| ペット詳細 `GET /api/pets/:petId` | < 400ms | < 1000ms | < 2000ms | High |
| 緊急情報更新 `PUT /api/pets/:petId/emergency-info` | < 500ms | < 1000ms | < 2000ms | High |
| 健康データ取得 `GET /api/pets/:petId/health/*` | < 500ms | < 1500ms | < 3000ms | Medium |
| OCR抽出 `POST /api/pets/:petId/medical-documents/:id/extract` | < 10s | < 30s | < 60s | Low |

### ページロード時間

| 画面 | First Contentful Paint | Time to Interactive | 条件 |
|---|---|---|---|
| 緊急公開画面 `/e/:token` | < 1.0s | < 2.0s | 3G回線 |
| ペット一覧 `/pets` | < 1.5s | < 3.0s | 4G回線 |
| ペット詳細 `/pets/:petId` | < 2.0s | < 3.5s | 4G回線 |
| その他画面 | < 2.0s | < 4.0s | 4G回線 |

**重要**: 緊急公開画面は3G回線でも2秒以内に表示できることを保証する。

### データベースクエリ目標

```
単一テーブル参照: < 10ms
JOIN（2-3テーブル）: < 50ms
集計クエリ（健康トレンド）: < 200ms
```

### パフォーマンス最適化戦略

```
フロントエンド:
  - Next.js 静的生成・ISR（緊急公開画面は除く）
  - 画像最適化: next/image
  - コード分割: 動的インポート
  - フォントのプリロード

バックエンド:
  - Prisma クエリの select 最適化（必要フィールドのみ）
  - N+1 クエリの排除（include / select 適切化）
  - 検索インデックス（householdId, petId, createdAt）

キャッシュ戦略:
  - ペット一覧: SWR (stale-while-revalidate)
  - 緊急情報: キャッシュなし（常に最新を取得）
  - 静的コンテンツ: Vercel CDN キャッシュ
```

## 2. 可用性・信頼性要件

### 可用性目標

```
サービス全体: 99.9%（月次ダウンタイム: < 43分）
緊急公開API: 99.95%（月次ダウンタイム: < 22分）
```

**理由**: 緊急公開APIは救急時に利用されるため、より高い可用性が必要。

### 依存サービスのSLA

| サービス | 公式SLA | 当サービスへの影響 |
|---|---|---|
| Vercel（ホスティング） | 99.99% | 全機能 |
| Supabase（DB/Auth） | 99.9%（有料プラン） | 全機能 |
| Supabase Storage | 99.9% | 写真機能 |
| Stripe | 99.99% | 課金機能のみ |

### 障害対応方針

**単一障害点の排除**:
- Vercel はグローバル CDN で冗長化
- Supabase は高可用性設定（有料プラン）

**フォールバック設計**:
```
緊急公開API:
  - Supabase RPC が失敗 → 500エラー（改善: リードレプリカへのフォールバック）
  
写真アップロード:
  - bucket未作成時は自動作成してリトライ（TASK-183で実装済み）

課金チェック:
  - Stripe API がタイムアウト → エラーハンドリングで画面を表示（TASK-197で対応）
```

## 3. スケーラビリティ要件

### 想定スケール（Phase 1）

```
ユーザー数: ~10,000 ユーザー
ペット数: ~50,000 頭
月次APIリクエスト: ~5,000,000 リクエスト
ストレージ: ~500GB（写真・書類）
```

### スケールアップポイント

| 規模 | 対応 |
|---|---|
| ~10,000 ユーザー | 現在のVercel + Supabase で対応可能 |
| ~100,000 ユーザー | Supabase の接続プール設定・クエリ最適化 |
| ~1,000,000 ユーザー | CDN 強化・Redis キャッシュ・読み取りレプリカ |

### データ増加への対応

```
写真・書類:
  - Supabase Storage のオブジェクトストレージはほぼ無制限
  - コスト: $0.021/GB（Supabase）
  
DB レコード:
  - 年間推定増加: 1ユーザー × 100記録/年 = 100万レコード（10,000ユーザー）
  - PostgreSQL は数十億レコードまで対応可能（適切なインデックスが前提）
  
バックアップ:
  - ストレージ増加に比例してバックアップコストも増加
  - 3ヶ月以上前の医療書類を Cold Storage に移動（コスト削減）
```

## 4. 保守性・拡張性要件

### コード品質要件

```
TypeScript:
  - strict: true（全ての型チェックを有効化）
  - any 型の使用禁止
  - 型推論を活用し、型注釈を最小化

関数設計:
  - 関数は最大50行以内
  - 循環複雑度: 10以下
  - 単一責任の原則

ファイルサイズ:
  - 最大400行（AGENTS.md に定義）
  - 超過時は機能分割
```

### テストカバレッジ要件

```
ユニットテスト: 80%以上
統合テスト: 主要APIエンドポイントを全てカバー
E2Eテスト: クリティカルユーザーフローを全てカバー
```

### 拡張性の設計

**動物病院連携（将来機能）**:
```
設計原則: 既存データモデルを壊さずに拡張
方法: 
  - 新テーブル（VeterinaryClinic, ClinicPartner 等）を追加
  - 既存の Pet.vetName/vetPhone を参照として維持
  - 将来的に ClinicId への外部キーに移行
```

**多言語対応（将来機能）**:
```
設計原則: i18n 対応を考慮したコード設計
方法:
  - next-intl または next-i18next の採用
  - ハードコードされた日本語文字列を翻訳キーに移行
  - 緊急公開画面は Accept-Language ヘッダーで言語切替
```

## 5. ユーザビリティ要件

### 操作性

```
タッチターゲット: 最小44px × 44px (Apple HIG準拠)
緊急操作: 3タップ以内で QRコードに到達
入力補助: オートコンプリート、日本語入力対応
エラー回復: 操作を中断してもデータが失われない
```

### 学習容易性

```
初回利用: オンボーディング（3ステップ）で最重要機能を体験
コンテキストヘルプ: ?(クエスチョンアイコン)で機能説明（TASK-195）
空状態: 必ずCTAと説明を表示（TASK-162）
```

### 通知設計

```
投薬リマインダー: 設定した時刻にメール/プッシュ通知
ワクチン期限: 30日前・14日前・7日前に通知（未実装）
トライアル終了: 7日前・3日前・当日に通知（未実装）
サービス障害: ステータスページで告知
```
