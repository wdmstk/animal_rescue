## 3. ユースケース詳細

### UC-05: QRコード生成・表示（最重要）

**目的**: 緊急時に3タップ以内でQRコードを表示し、医療従事者に情報提供する

**事前条件**:
- ログイン済み
- 対象ペットが登録済み
- 緊急情報が入力済み

**主フロー**:
1. ユーザーがペット一覧画面を開く
2. 対象ペットをタップ
3. ペット詳細画面で「緊急QR」ボタンをタップ
4. QRコードが全画面表示される
5. 医療従事者がスマートフォンでスキャン
6. `/e/:token` 画面が開き、緊急情報が表示される

**代替フロー**:
- 3a. QRコードが未生成の場合: 自動生成してから表示
- 6a. トークンが無効な場合: 404画面を表示

**事後条件**:
- 緊急情報が医療従事者の端末で閲覧可能な状態になる

**設計上の制約**:
- 3タップ以内での到達を保証する
- 画面輝度を最大にする（パニック状態での可視性確保）
- QRコードのサイズを十分大きく表示する（最低250px×250px）

**現状の問題点**:
- QRボタンがペット詳細画面の下部にある（スクロールが必要）
- **改善案**: ペット詳細画面の最上部にスティッキーな「緊急QR」ボタンを固定

---

### UC-06: 緊急情報閲覧（医療従事者側）

**目的**: 認証なしで、最小限の操作で必要な医療情報を確認する

**アクター**: 救急獣医師、夜間病院スタッフ、ペットホテルスタッフ

**事前条件**:
- 飼い主がQRコードを表示している
- QRトークンが有効（isActive=true）

**主フロー**:
1. スマートフォンカメラでQRコードをスキャン
2. ブラウザが起動し、`/e/:token` が開く
3. ページが即座に読み込まれる（<2秒）
4. 緊急情報カードが表示される:
   - 動物名・種類・品種・年齢・体重
   - 持病・既往歴（赤いバッジで強調）
   - アレルギー（赤いバッジで強調）
   - 現在の投薬情報
   - かかりつけ病院（電話番号ワンタップ）
   - 緊急連絡先（電話番号ワンタップ）
5. 必要に応じてワンタップで電話発信

**設計要件**:
- ページ読み込み時間: <2秒（3G回線でも）
- 文字サイズ: 最低16px以上（緊急閲覧向け）
- コントラスト比: WCAG AA準拠
- ページを印刷しても情報が正しく表示される

## 4. 業務フロー詳細

### オンボーディングフロー（初回ユーザー）

```
ステップ1: アカウント作成
  入力: メールアドレス・パスワード
  処理: Supabase Auth でユーザー作成 → 世帯自動作成
  完了条件: ログイン状態になる

ステップ2: ペット登録（必須）
  入力: ペット名・種類・品種・性別・誕生日
  完了条件: ペット1頭以上が登録される

ステップ3: 緊急情報入力（推奨）
  入力: 持病・アレルギー・かかりつけ病院・緊急連絡先
  完了条件: PetEmergencyInfo レコードが存在する

ステップ4: QR確認（推奨）
  操作: QRコードを確認・テストスキャン
  完了条件: PetEmergencyToken が Active 状態になる

→ ダッシュボードへ
→ 30日後: 課金導線（Stripe Checkout）
```

### 救急対応フロー（最重要）

```
状況: 深夜、ペットが急に具合が悪くなった

T+0:00  飼い主がペットの異変に気づく
T+0:30  スマートフォンを取り出す（震えている）
T+0:45  アプリを起動（生体認証ロック解除）
T+1:00  ペット一覧でペットをタップ（タップ2回目）
T+1:15  緊急QRボタンをタップ（タップ3回目）
T+1:20  QRコードが全画面表示

T+5:00  夜間救急病院に到着
T+5:30  受付スタッフがQRをスキャン
T+5:45  緊急情報画面が表示（認証不要）
T+6:00  持病・投薬情報を確認
T+6:15  治療方針の決定が始まる

この75秒以内での情報提供が目標
```

### 日常ケアフロー（毎日）

```
朝: 投薬リマインダー受信
  → アプリを開く（通知タップ）
  → 投薬記録にチェック
  → 30秒で完了

週次: 体重記録
  → ペット詳細を開く
  → 健康記録タブ
  → 体重を入力
  → グラフで推移確認

月次: 病院受診後
  → 医療記録タブ
  → 「+」ボタンで記録追加
  → 書類写真をOCR読取
  → テキストを確認・修正
  → 保存
```

## 5. 制約条件

### 技術的制約
- Next.js App Router の制約（サーバーコンポーネント/クライアントコンポーネント）
- Supabase無料プランの制限（本番は有料プランが必要）
- Vercelの無料プランの制限

### 法的制約
- **医療行為禁止**: 診断・治療行為を提供しているとみなされる機能は禁止
- **個人情報保護法**: ペット情報も個人情報に準じた取り扱いが必要
- **特定商取引法**: 月額課金サービスとして表記義務あり

### UX制約
- **3タップルール**: 緊急情報へのアクセスは最大3タップ以内
- **オフライン対応**: 通信障害時でもQR画面は表示できること
- **文字サイズ**: 緊急画面は最低16px（できれば18px以上）

### セキュリティ制約
- QRトークンは推測困難なUUIDを使用
- 公開URLには内部ペットIDを露出しない
- RLSによるテナント分離を必須とする

## 6. 前提条件

### 環境前提
- スマートフォン（iOS / Android）での利用を主とする
- 画面サイズ: 375px以上（iPhone SE以上）を基準
- 日本語のみ対応（Phase 1）

### ユーザー前提
- スマートフォンの基本操作ができる
- QRコードの使い方を知っている（または説明があれば理解できる）
- 月額680円の課金に同意している

### サービス前提
- 30日の無料トライアル後、月額680円の課金が必要
- 世帯単位での課金（家族メンバーは追加料金なし）
- ペット頭数の制限なし

### 医療免責前提
- 本サービスは医療情報の記録・共有ツールであり、診断・治療を提供しない
- 緊急時は必ず獣医師の判断に従うこと
- 情報の正確性は飼い主が管理する責任を持つ

## 改善提案

### 現状の問題点と改善案

#### 問題1: 緊急情報の項目が不足
**現状**: `disease`（フリーテキスト）、`allergy`（フリーテキスト）のみ
**問題**: 構造化されていないため、緊急時に必要な情報が埋もれる
**改善案**:
```
- 持病: タグ入力（チェックボックス + 自由入力）
- アレルギー: タグ入力（薬剤・食品・その他を分類）
- 血液型: ドロップダウン
- 夜間救急病院: 別フィールドで病院名・電話・住所
```

#### 問題2: 緊急QRへのアクセスが深い
**現状**: ペット詳細画面をスクロールしてQRボタンを探す必要あり
**改善案**:
- ペット詳細画面のヘッダー直下に「緊急QR」ボタンを固定配置（スティッキー）
- ペット一覧画面で各ペットカードに「緊急」ボタンを表示
- アプリのホーム画面に「緊急ショートカット」ウィジェット

#### 問題3: 緊急公開画面の情報優先度
**現状**: 情報が並列に表示されている
**改善案**:
```
ファーストビュー（スクロール不要）:
  1. 動物名・写真
  2. 持病（赤色強調）
  3. アレルギー（赤色強調）
  4. 現在の薬
  5. 緊急連絡先（大きな発信ボタン）

スクロール後:
  6. かかりつけ病院
  7. ワクチン状況
  8. 直近の医療記録
```
