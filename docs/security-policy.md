# Security Policy

## 基本方針
- 最小権限
- テナント分離（RLS）
- 秘密情報の安全管理
- 監査可能性

## 禁止事項
- 秘密鍵をリポジトリへコミット
- クライアントで service role key を利用
- 認可チェックなしで業務データへアクセス

## 必須事項
- すべての保護APIで認証確認
- household_id / pet_id の所有境界チェック
- 入力値バリデーション
- エラーメッセージで機微情報を漏らさない

## 実装ガードレール
- `SUPABASE_SERVICE_ROLE_KEY` はサーバー側のみで利用する
- 公開APIは最小情報のみ返却し、内部識別子を露出しない
- API Routeでは UUID / 文字列長 / 必須項目を先に検証してからDBアクセスする
- RLS定義変更時は `supabase/migrations/*_rls.sql` と実装差分を同時レビューする

## 定期確認
1. 四半期ごとに秘密情報のローテーションを確認
2. 主要APIの認可テスト（unit/integration）を実行
3. SupabaseのRLSポリシー差分をレビュー

## インシデント対応
1. 影響範囲の特定
2. キーのローテーション
3. 一時対策と恒久対策の分離
4. 再発防止策の記録
